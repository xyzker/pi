Session Management Architecture

packages/agent/src/harness/session/ — 7 files that implement the session tree, storage, and repository layer for the pi Agent Harness.

1. Overview

The session system manages conversation state for AI coding agents. It stores every user message, assistant reply, tool output, model change, and metadata as a tree of entries — not a flat linear transcript. This tree enables branching (forking to explore alternatives), compaction (summarizing old context), and navigation between branches.

Storage Two Backends

JSONL files on disk (jsonl-storage.ts) for production persistence, and in-memory maps (memory-storage.ts) for fast tests.

API Repository Pattern

SessionRepo manages collections of sessions (CRUD + fork). Session wraps a SessionStorage to provide a typed append API.

Core Tree Structure

Every entry has a parentId. A leaf pointer tracks the current tip. Branching just records a new LeafEntry pointing elsewhere.

3. Data Model: The Entry Tree

Every session is a tree, not a linear log. Each node is a SessionTreeEntry with id, parentId, timestamp, and a type discriminator. The leaf pointer tracks the current active tip; branching writes a new LeafEntry pointing to a different node.

Key insight: Leaf entries create branches. The "current position" in the tree is simply the last leaf's targetId. To switch branches, write a new LeafEntry with a different targetId. The old messages are never deleted — you can always navigate back.

4. UUID v7 (uuid.ts)

UUID v7 embeds a 48-bit Unix timestamp (ms) in the first 6 bytes, making IDs lexicographically sortable by creation time. A monotonic sequence ensures uniqueness within the same millisecond. The remaining bits use crypto-grade randomness.

019a8c3e- 5f1a- 7a3b- 82c1- d4e5f6a7b8c9
█ 48-bit timestamp (ms) █ Version (0x7) █ 12+ bits monotonic sequence █ Random bits

Monotonic sequence logic

// Same millisecond? Increment the 28-bit sequence counter.
// Sequence overflow (very rare)? Increment the timestamp instead.
if (timestamp > lastTimestamp) {
    sequence = random[6..9] as 28-bit integer;  // random seed for new ms
    lastTimestamp = timestamp;
} else {
    sequence = (sequence + 1) >>> 0;             // wraparound safe
    if (sequence === 0) lastTimestamp++;          // overflow → bump time
}

Entry IDs only use the first 8 chars of the UUID (via generateEntryId) for compactness, falling back to the full UUID after 100 collisions.

5. SessionStorage

The SessionStorage interface is the low-level contract that both implementations fulfill:

interface SessionStorage<TMetadata> {
  getMetadata():       Promise<TMetadata>;             // id, createdAt, (cwd, path for JSONL)
  getLeafId():         Promise<string | null>;          // current tip of the branch
  setLeafId(id):       Promise<void>;                   // writes a LeafEntry, moving tip
  createEntryId():     Promise<string>;                 // generate a unique entry id
  appendEntry(entry):  Promise<void>;                   // add one entry to the tree
  getEntry(id):        Promise<SessionTreeEntry | undefined>;
  findEntries(type):   Promise<SessionTreeEntry[]>;     // filter by entry type
  getLabel(id):        Promise<string | undefined>;      // label for a given entry id
  getPathToRoot(leaf): Promise<SessionTreeEntry[]>;      // walk parentId chain up from leaf
  getEntries():        Promise<SessionTreeEntry[]>;      // all entries
}

impl: JsonlSessionStorage

Persists every entry as a single JSON line in a .jsonl file. On open, it reads the entire file, validates the header (version 3), parses all entry lines, and builds byId and labelsById caches. On append, it writes the line to disk and updates caches.

Tradeoff: reads whole file on open, but writes are O(1) appends. The byId map is rebuilt from scratch on each open.

impl: InMemorySessionStorage

Stores entries in a plain SessionTreeEntry[] with a Map<string, SessionTreeEntry> for O(1) lookups. Constructor can optionally seed from existing entries (used during fork). No disk I/O — used exclusively in tests and the harness test suite.

Shared logic (duplicated between implementations)

Both storage files contain identical copies of these private helpers:

  • updateLabelCache() / buildLabelsById() — maintain a targetId → label map from LabelEntry nodes
  • generateEntryId() — tries truncated 8-char UUID v7, falls back to full UUID after 100 collisions
  • leafIdAfterEntry() — for LeafEntry, return targetId; otherwise return the entry's own id

The duplication is intentional to avoid coupling the two backends. The JSONL backend also has header parsing (parseHeaderLine) and entry validation (parseEntryLine) which are file-format-specific.

6. Session Class (session.ts)

Session wraps a SessionStorage and provides typed append methods that automatically fill in id, parentId, and timestamp from the current leaf. It's the main API used by the harness.

// Core pattern: every append method follows this shape
async appendMessage(message: AgentMessage): Promise<string> {
  return this.appendTypedEntry({
    type: "message",
    id:     await this.storage.createEntryId(),    // auto-generated
    parentId: await this.storage.getLeafId(),       // current tip
    timestamp: new Date().toISOString(),
    message,
  } satisfies MessageEntry);
}

// Same pattern for: appendThinkingLevelChange, appendModelChange,
//   appendCompaction, appendCustomEntry, appendLabel, appendSessionName,
//   appendCustomMessageEntry

moveTo() — Branch Switching

async moveTo(entryId: string | null, summary?): Promise<string | undefined> {
  await this.storage.setLeafId(entryId);    // writes a LeafEntry(targetId: entryId)
  if (summary) {
    // Optionally write a BranchSummaryEntry explaining the move
    return this.appendTypedEntry({ type: "branch_summary", ... });
  }
}

This is all there is to branching: call moveTo(someEntryId) and the leaf pointer jumps. A BranchSummaryEntry optionally records why you moved, so the context can include an explanation for the model.

buildContext() — Tree → Chat Messages

async buildContext(): Promise<SessionContext> {
  return buildSessionContext(await this.getBranch());
  // getBranch() calls storage.getPathToRoot(leafId)
  // buildSessionContext() walks the path from root → leaf,
  //   collecting messages, model/thinking state, and handling compaction
}

getBranch() calls getPathToRoot(currentLeafId), which walks parentId pointers up to the root, returning entries in chronological order. buildSessionContext() then:

  1. Scans the path for the most recent thinking_level_change, model_change, and assistant message (for model info)
  2. If a compaction entry exists, builds the context as: compaction summary → everything after firstKeptEntryId
  3. If no compaction, includes all messages, custom_message entries, and branch_summary entries from the full path

7. SessionRepo

JsonlSessionRepo

Organizes .jsonl files in a directory tree:

~/.pi/sessions/
  --home-kexi-projects-myapp--/
    2026-05-21T10-30-00-000Z_abc123.jsonl
    2026-05-21T11-15-00-000Z_def456.jsonl
  --home-kexi-projects-other--/
    ...

Directory names encode the cwd (e.g., /home/kexi/projects/myapp → --home-kexi-projects-myapp--). Session files are named <timestamp>_<uuid>.jsonl.

list() scans all session dirs, reads only the first line of each .jsonl file (the header) to extract metadata, and sorts by createdAt descending.

InMemorySessionRepo

A trivial Map<string, Session>. Used in tests where no filesystem is available or desired.

8. JSONL File Format

Each .jsonl session file contains one JSON object per line. Line 0 is a session header. All subsequent lines are tree entries.

// Line 0 — Session Header
{"type":"session","version":3,"id":"abc123...","timestamp":"2026-05-21T10:30:00.000Z","cwd":"/home/user/project","parentSession":"/path/to/parent-session.jsonl"}

// Line 1 — First message (user)
{"type":"message","id":"a1b2c3d4","parentId":null,"timestamp":"2026-05-21T10:30:01.000Z","message":{"role":"user","content":[{"type":"text","text":"Write a function"}]}}

// Line 2 — Assistant reply
{"type":"message","id":"e5f6g7h8","parentId":"a1b2c3d4","timestamp":"2026-05-21T10:30:05.000Z","message":{"role":"assistant","content":[...],"provider":"anthropic","model":"claude-sonnet-4-20250514"}}

// Line 3 — Leaf pointer (records current tip)
{"type":"leaf","id":"i9j0k1l2","parentId":"e5f6g7h8","timestamp":"2026-05-21T10:30:05.500Z","targetId":"e5f6g7h8"}

// Line 4 — Compaction (old messages summarized)
{"type":"compaction","id":"m3n4o5p6","parentId":"e5f6g7h8","timestamp":"...","summary":"User asked for a function...","firstKeptEntryId":"i9j0k1l2","tokensBefore":4500}

// Line 5 — Branch switch (moved to a different point)
{"type":"branch_summary","id":"q7r8s9t0","parentId":"i9j0k1l2","timestamp":"...","fromId":"i9j0k1l2","summary":"Moved to fix the bug..."}

Header Validation

On open, parseHeaderLine() validates: type === "session", version === 3, id is a non-empty string, timestamp is non-empty, cwd is non-empty. Failures throw SessionError("invalid_session").

loadJsonlSessionMetadata() reads only line 0 — this is what list() uses to scan all sessions efficiently.

9. Branching & Forking

Branching within a session (lightweight)

// Session.moveTo() writes a LeafEntry with a different targetId
await session.moveTo(someEntryId);
// Future appendMessage() calls now extend from someEntryId

The old path is never removed. You can always moveTo() back. This is how the session tree navigator works.

Forking (creating a new session from an existing one)

// repo-utils.ts:
export async function getEntriesToFork(storage, options): Promise<SessionTreeEntry[]> {
  if (!options.entryId) return storage.getEntries();          // fork entire session
  const target = await storage.getEntry(options.entryId);
  if (options.position === "at") {
    effectiveLeafId = target.id;                               // fork AT entry (include it)
  } else {
    // "before": only fork entries before this user message
    effectiveLeafId = target.parentId;                         // fork BEFORE entry
  }
  return storage.getPathToRoot(effectiveLeafId);               // only the ancestors
}

Fork Flow

  1. repo.fork(sourceMetadata, { entryId?, position?, id? })
  2. Open the source session, call getEntriesToFork() to compute which entries to copy
  3. Create a new .jsonl file with a fresh header (optionally linked via parentSessionPath)
  4. Copy each entry from the source into the new session
  5. Return the new Session ready for new appends

10. Context Compaction

When a conversation grows too large for the model's context window, the harness inserts a CompactionEntry. The compaction entry stores:

  • summary — LLM-generated summary of old messages
  • firstKeptEntryId — the first entry not summarized (i.e., where the "live" context resumes)
  • tokensBefore — token count before compaction

buildSessionContext() handles this in session.ts:

if (compaction) {
  // 1. Prepend the compaction summary as a system/user message
  messages.push(createCompactionSummaryMessage(compaction.summary, ...));

  // 2. Find the compaction entry's position in the path
  // 3. Skip everything BEFORE firstKeptEntryId (already summarized)
  // 4. Include everything AFTER firstKeptEntryId (still relevant)
  // 5. Include everything that came AFTER the compaction entry (new messages)
}

Multiple compactions are supported — each one only keeps what came after its firstKeptEntryId. The logic walks the path finding the first compaction and applies the windowing from there.

12. Summary

Key Design Decisions

  • Tree, not log. Sessions encode branching via LeafEntry nodes, enabling infinite undo/redo and parallel exploration.
  • Repository + Storage split. SessionRepo manages collections; SessionStorage manages individual session trees. Clean separation of concerns.
  • Two backends, one interface. JSONL for production, in-memory for testing. Both implement SessionStorage identically.
  • Compaction is a first-class node type. Instead of modifying existing entries, compaction inserts a CompactionEntry that buildSessionContext() interprets as a window boundary.
  • UUID v7 for sortable IDs. Time-ordered UUIDs mean chronological sorting is just string comparison, with monotonic sequence for same-ms uniqueness.
