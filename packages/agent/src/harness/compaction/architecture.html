Compaction Subsystem

The compaction/ directory implements context-window management for the pi coding agent. When an LLM conversation grows beyond the model's token budget, this subsystem compresses older message history into structured summaries that preserve the semantic state while reclaiming context-window capacity.

Three files, three responsibilities: compaction.ts — main compaction engine (cut points, summary generation, split turns)
branch-summarization.ts — summarizes abandoned conversation branches
utils.ts — shared helpers: file-operation tracking and message serialization

File Overview

compaction.ts TS

The main engine. Determines when to compact, where to cut, and how to summarize.

  • Token estimation with provider usage anchoring
  • Cut-point selection respecting turn boundaries
  • LLM-driven summary generation (initial + iterative)
  • Split-turn handling when compaction happens mid-turn

branch-summarization.ts TS

Handles the case where a user navigates away from a conversation branch.

  • Collects entries on the abandoned branch path
  • Prepares messages within a token budget
  • Generates a structured branch summary
  • Propagates file-operation metadata from child branches

utils.ts TS

Shared data structures and serialization logic used by both modules.

  • FileOperations type — tracks read/written/edited files
  • extractFileOpsFromMessage — parses tool calls
  • serializeConversation — LLM messages → plain text
  • computeFileLists / formatFileOperations — sorted XML output

Key Design Decisions

Hybrid Token Estimation

estimateContextTokens() anchors on the last successful Usage block from the provider, then adds heuristically-estimated tokens for messages that follow. This gives more accurate context-window tracking than pure character-based heuristics while still working when provider usage data is unavailable.

Turn-Aware Cut Points

findCutPoint() avoids cutting in the middle of tool-result sequences by only considering "user-visible" entry boundaries (user messages, assistant replies, branch summaries). findValidCutPoints() explicitly excludes toolResult entries. When the optimal cut falls mid-turn, the system enters split-turn mode: the turn prefix is summarized separately so the retained suffix has context.

Iterative Summarization

generateSummary() accepts an optional previousSummary. When present, it uses UPDATE_SUMMARIZATION_PROMPT to merge new conversation messages into the existing summary rather than regenerating from scratch. This is critical for sessions that compact multiple times — each compaction builds on the last.

File-Operation Metadata

Every compaction and branch summary records which files were read and modified. This metadata is stored in CompactionDetails / BranchSummaryDetails and propagated forward so the agent always knows which files are relevant, even after the original messages are compacted away.

Summarization Prompt Format

Both compaction and branch summarization generate structured summaries using this template:

## Goal
[What is the user trying to accomplish?]

## Constraints & Preferences
- [Requirements mentioned by user]

## Progress
### Done
- [x] [Completed tasks]

### In Progress
- [ ] [Current work]

### Blocked
- [Issues preventing progress]

## Key Decisions
- **[Decision]**: [Brief rationale]

## Next Steps
1. [Ordered list]

## Critical Context
- [Data, examples, references needed to continue]

<read-files>
path/to/file1.ts
path/to/file2.ts
</read-files>

<modified-files>
path/to/file3.ts
</modified-files>
Branch summaries use a slightly different template with a preamble: "The user explored a different conversation branch before returning here." The Critical Context section is omitted from branch summaries; they use Constraints & Preferences instead.

Session Tree Integration

Compaction entries are stored in the session tree as CompactionEntry nodes with type "compaction". They contain:

  • summary — the LLM-generated structured summary text
  • firstKeptEntryId — UUID of the first entry retained after this compaction
  • tokensBefore — estimated context tokens before compaction occurred
  • details: CompactionDetails — { readFiles, modifiedFiles }

When building session context (buildSessionContext), entries before firstKeptEntryId are replaced by the compaction summary. The summary is injected as a compactionSummary-role message.

Error Handling

Both modules use a Result<T, E> pattern (via ok() / err()) with typed error classes:

  • CompactionError — codes: "aborted", "summarization_failed", "invalid_session"
  • BranchSummaryError — codes: "aborted", "summarization_failed"

Aborted/error stop reasons from completeSimple are explicitly surfaced rather than silently swallowed.

Entry-to-Message Conversion

getMessageFromEntry() converts a SessionTreeEntry into an AgentMessage. Both compaction.ts and branch-summarization.ts implement their own versions with slightly different handling:

The key difference: getMessageFromEntry() in branch-summarization excludes toolResult messages because those are not useful for branch summaries. Compaction's getMessageFromEntryForCompaction() excludes compaction entries themselves to avoid recursive summary nesting.
